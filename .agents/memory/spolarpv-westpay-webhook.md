The WestPay deposit webhook (`POST /api/webhook/westpay` in `server/routes.ts`) verifies HMAC signatures using a secret resolved as `settings.westpayWebhookSecret` (platform_settings table, editable via Admin > Paramètres > WestPay) first, falling back to the `WESTPAY_WEBHOOK_SECRET` env var.

**Why:** the production deployment runs on an external Plesk server the agent/dev has no shell access to. Setting env vars there requires the user to do it manually and redeploy; storing the secret as a DB setting lets it be configured/rotated from the admin UI without touching Plesk. Same pattern already existed for `omnipayCallbackKey`.

**How to apply:** if WestPay (or similar gateway) deposits aren't auto-crediting, check first whether `westpayWebhookSecret` is set in `platform_settings` (or the env var as fallback) — a missing secret makes the webhook handler silently no-op (still returns 200 to the gateway, so it looks like delivery "succeeded" but nothing is credited). Also: the public `GET /api/settings` route strips `westpayWebhookSecret` and `omnipayCallbackKey` before responding — keep any new secret-like setting key added there excluded too.

Correction (verified 2026-07-15): this Replit workspace's `SUPABASE_DATABASE_URL` is a SEPARATE dev/test database, NOT shared with the Plesk production deployment — confirmed by querying for a real user known to exist only in prod and finding 0 rows. Products/tasks/settings look similar only because both DBs were seeded from the same `seed.ts` defaults, not because they're the same instance. Do not assume DB-level changes made from this workspace affect production; verify independently (e.g. via curl against the live prod domain) before treating a DB write as having fixed anything live.

## Deposit-matching fallback caused a real mis-credit — fixed 2026-07-15

`storage.findProcessingWestpayDeposit(amount, payerPhone)` used to fall back to FIFO (oldest processing deposit of that amount) whenever the payer phone didn't match any candidate's stored `accountNumber` — even when a payer phone WAS given but simply didn't match anyone on file. In production this credited a *different* user's unrelated same-amount deposit instead of leaving it unmatched, verified with a live test webhook (200 XOF test credited the super-admin's own pending 200 deposit instead of the real depositor's).

**Why:** guessing is unsafe when money is involved — the safe default when a payer phone doesn't match anyone is to NOT auto-approve, not to pick the closest thing.

**How to apply:** the fixed rule (in `server/storage.ts`) only falls back to FIFO when either (a) the gateway didn't supply a payer phone at all, or (b) there is exactly one processing candidate for that amount (so amount alone is unambiguous). If a payer phone is supplied and doesn't match, and there are 2+ same-amount candidates, the deposit is left unmatched (requires manual admin review) rather than guessing. Apply the same caution to any other gateway matching logic (Soleaspay/OmniPay) if it has a similar blind FIFO fallback.

Also fixed same day: `approveWestpayDeposit` now overwrites the deposit's `accountNumber` with the real payer phone reported by the webhook (previously it kept showing the account owner's registered phone, not the actual mobile-money number used to pay — confusing in the admin panel).

Status: verified fixed and confirmed working end-to-end in production (Plesk) on 2026-07-15 — signature check, deposit matching, balance crediting, and payer-number display all confirmed via a live test deposit.

**CORRECTION (2026-07-15, same day):** the earlier claim in this file that the Replit workspace dev DB is separate from the Plesk production DB was WRONG. Querying `SUPABASE_DATABASE_URL` from this workspace shows real, live-timestamped user/deposit rows matching what the user reports doing in the app in real time (new users, new deposits appearing seconds after the user acts). **The dev DB and production DB are the same Supabase instance.** Treat any DB write from this workspace as a live production write — do not casually insert/update/delete test data here.

**Update 2026-07-15:** added a `webhook_logs` table (+ `webhookLogs` in `shared/schema.ts`) that now records every inbound call to `/api/webhook/westpay` (outcome, signature-valid flag, a whitelisted safe subset of headers, body) — so future sessions CAN check the DB directly to see whether WestPay ever actually called the server for a given deposit, instead of relying only on the WestPay merchant dashboard.

**Root cause confirmed 2026-07-15 for a stuck deposit:** the DB-stored `westpayWebhookSecret` didn't match WestPay's dashboard-displayed secret at the exact time a real deposit's payment was confirmed; WestPay called the webhook, got a signature-verification rejection, and did not retry afterward (no visible retry/backoff). Once the DB secret was corrected, new signed calls verified successfully immediately (no redeploy needed, confirmed via `platform_settings.modified_at`). **Lesson:** if a real deposit is stuck in "processing" with `webhook_logs` empty for it, check whether the DB secret was wrong *at the time the deposit was made* (compare deposit `created_at` vs `platform_settings.modified_at` for the secret) — WestPay appears to give up after one failed attempt rather than retrying, so fixing the secret later does not un-stick deposits made before the fix; those need manual admin approval once WestPay confirms (on their own dashboard) that the money was actually received.

**CRITICAL SECURITY FINDING 2026-07-15:** the production Plesk/nginx/Passenger reverse-proxy setup leaks ALL server env vars (SESSION_SECRET, full DB connection string with password, all WESTPAY_KEY_* per-country API keys, WESTPAY_WEBHOOK_SECRET) into inbound HTTP request headers under a key literally named `!~passenger-envvars` (base64-encoded, null-byte-separated `KEY\0VALUE\0KEY\0VALUE...`), on at least the `/api/webhook/westpay` route — confirmed by a real WestPay "Test webhook" dashboard call. This is a hosting-level nginx/Passenger misconfiguration, not an app bug, and is outside this app's control to fully fix.
**Why it matters:** any code that logs/echoes/stores full `req.headers` on this app is at risk of persisting live secrets in plaintext. One diagnostic webhook_logs row briefly did before it was caught and redacted the same session.
**How to apply:** never log/store raw `req.headers` anywhere in this app — always use an explicit whitelist. `server/index.ts` now has a global middleware (added 2026-07-15, first thing after `express()` init) that strips any header key starting with `!~` or containing `passenger-envvars` before it reaches any route/logging, and warns server-console-only (never persisted) when it fires — check server logs for that warning to know if the hosting misconfig is still active. The user was advised to rotate SESSION_SECRET, the Supabase DB password, and all WestPay API keys/webhook secret, and to report the nginx/Passenger config issue to their Plesk host — confirm whether that rotation happened before trusting any of those values are still secret.

**Priority flipped 2026-07-15 (explicit user request):** `process.env.WESTPAY_WEBHOOK_SECRET` (set directly on Plesk) is now checked BEFORE the `platform_settings.westpayWebhookSecret` DB value, reversing the original design. **Why:** the DB value and the real WestPay-dashboard secret drifted out of sync at least twice, silently breaking the webhook each time (rejected with no retry from WestPay); the user wants Plesk's own env var to be the single source of truth they control directly, with the DB value only as a dev-environment fallback (this Replit workspace has no such env var, so it still reads the DB row). **How to apply:** if a deposit is stuck again, check the current effective secret's origin (env var vs DB) before assuming the DB row is authoritative — and remember this is a *code* change (`server/routes.ts`), not just a DB row, so it only takes effect on Plesk once that server pulls and rebuilds this exact commit.
